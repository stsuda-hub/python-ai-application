# Streamlit でつくる「AIライティングツール」のホーム画面を読み解く

こんにちは。今日はあなたのプロジェクトにある `app.py` を題材に、**Streamlit アプリのホーム画面がどう作られているか**を一緒に読み解いていきます。わずか 44 行の短いコードですが、Streamlit の基本エッセンスがぎゅっと詰まっています。

## この `app.py` の役割

結論から言うと、このファイルは **「ランディングページ（玄関）」** です。

実際の生成処理（ブログ記事作成、翻訳、要約など）は `pages/` フォルダの中にあるファイルが担当しています。`app.py` 自体は **「どんなツールが使えますか？」「どう使い始めればいいですか？」をユーザーに伝えるだけ**のシンプルな役割に徹しています。

例えるなら、ホテルのロビー。客室（各ツール）には直接行かず、まずロビーで館内案内図を見るような構造です。

---

## コードを 4 つのブロックに分けて読む

### ① import と初期設定（1〜7行目）

```python
import streamlit as st

st.set_page_config(
    page_title="AIライティングツール",
    page_icon="✍️",
    layout="wide",
)
```

`streamlit` を `st` という短い名前で呼び出せるようにしています。これは Streamlit 界の "お約束" です。

`st.set_page_config` はブラウザタブ周りの設定です:

- `page_title` → ブラウザのタブに表示される文字
- `page_icon` → タブのアイコン（絵文字が使えます）
- `layout="wide"` → 画面を横いっぱいに広く使うレイアウト

**ポイント:** `set_page_config` は **スクリプトの最初の Streamlit 呼び出し**でなければならないルールがあります。他の `st.xxx` を呼んだ後に書くとエラーになります。

### ② タイトルと説明文（9〜10行目）

```python
st.title("AIライティングツール")
st.caption("Gemini API を使った個人用ライティングアシスタント")
```

- `st.title` → 画面上部の大きな見出し
- `st.caption` → その下の小さな補足テキスト

これだけで見た目の整ったヘッダーが完成します。HTML を書く必要はありません。

### ③ Markdown で案内表を表示（12〜36行目）

```python
st.markdown("""
    左のサイドバーから使いたいツールを選んでください。
    ...
""")
```

`st.markdown` は **Markdown 記法の文字列をそのまま HTML にレンダリング**してくれる関数です。

このブロックで使われている Markdown 記法:

| 記法 | 意味 |
| --- | --- |
| `### 利用できるツール` | 小見出し |
| `\| ツール \| 用途 \|` | 表（テーブル） |
| `1. ...` | 番号付きリスト |
| `` `.env` `` | コード風の装飾 |
| `[Google AI Studio](URL)` | リンク |

**ここが秀逸なポイント:** ユーザーが最初に迷うのは「何ができるアプリ？」「どう始めればいい？」の 2 つ。それを**ツール一覧表**と**セットアップ手順**の 2 セクションで即答しています。

### ④ サイドバーの説明（38〜43行目）

```python
with st.sidebar:
    st.header("このアプリについて")
    st.write("複数のAIライティング機能を...")
```

`with st.sidebar:` は「この `with` ブロックの中に書いた UI はすべてサイドバーに表示する」という意味です。

Streamlit のルールとして、`pages/` フォルダに置いたファイルは **自動的にサイドバーにメニューとして並びます**。つまり、このアプリではサイドバーに:

- （自動）各ページへのリンク一覧
- （手動）`with st.sidebar:` で書いた説明文

の 2 つが共存しています。

## なぜ `app.py` には生成処理が一切ないのか？

これは設計判断です。`CLAUDE.md` にも書かれているとおり、このプロジェクトは **3 層構成**を採用しています:

```
app.py             ← ホーム（案内のみ）
  ↓
pages/N_xxx.py     ← 各ツール（実際の処理）
  ↓
utils/gemini_client.py  ← Gemini API 呼び出し（共通部品）
```

もし `app.py` にブログ記事生成の処理を直接書いてしまうと、他のページを追加するたびにホーム画面が肥大化していきます。**玄関は玄関のまま**、処理は各ページに任せる。この分離のおかげで、新しいツールを追加したいときは `pages/` に 1 ファイル足すだけで済みます。

## まとめ: この 44 行から学べること

1. **`set_page_config` は最初に書く** — これは Streamlit の鉄則
2. **`st.markdown` は最強の時短ツール** — HTML を書かずに表もリンクも見出しも作れる
3. **`pages/` ディレクトリは自動でナビゲーションになる** — `app.py` でメニューを手書きする必要はない
4. **ホーム画面は "玄関" に徹する** — 処理を置かず、案内だけに集中することで保守性が上がる

Streamlit の良さは「Python だけで Web アプリが作れる」ことだとよく言われますが、本当の魅力は **"書かなくていいことが多い"** ことにあると私は思います。この `app.py` は、その思想をきれいに体現した好例です。
